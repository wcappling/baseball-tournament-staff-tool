# Architecture — Tournament IQ

## Overview

Single-process Python web app (FastAPI + SQLite) deployed on Railway. Serves both a browser SPA and a native iOS client from the same process.

```
Browser ──────────► FastAPI (app.py)
                     │
iOS client ─────────►│  /api/v1/* (bearer token)
                     │
                     ├── auth.py       session / token validation
                     ├── storage.py    SQLite (all reads/writes)
                     ├── services.py   refresh orchestration
                     ├── stats.py      W-L-T aggregation
                     └── collectors/   NCS · USSSA · Perfect Game
```

---

## Auth model

Two independent auth flows share the same middleware (`PasswordAuthMiddleware`).

### Web (browser)

1. POST `/login` with `team_slug` + `password` → sets `baseball_staff_session` cookie.
2. Cookie is `timestamp.team_id` signed with HMAC-SHA256 using `SESSION_SECRET`. Valid 30 days.
3. `get_web_team_id()` extracts `team_id` from every web request — falls back to `"default"` if no valid cookie (local dev without `STAFF_TOOL_PASSWORD` set).

Flags: `httponly=True`, `secure` in hosted mode, `samesite="lax"`.

### Native / iOS

1. POST `/api/v1/login` with JSON `{team_slug, password}` → returns `{team, session: {token, expires_at}}`.
2. Token is stored in the `team_sessions` table (SHA-256 hash of the raw token, 30-day expiry).
3. All subsequent `/api/v1/` requests send `Authorization: Bearer <token>`.
4. `_native_session()` dependency resolves the token and injects the session dict.

### Middleware bypass rules

The middleware lets requests through without credentials when:
- Auth is disabled (no `STAFF_TOOL_PASSWORD` set, not hosted)
- Path is public: `/login`, `/logout`, `/api/v1/login`, `/static/*`
- `DEV_AUTO_LOGIN=true` (dev only)
- Request is to `/api/v1/*` with a bearer token present (token validity is checked at the endpoint level)

## Multi-tenancy

All user data is scoped by `team_id` (a string, either `"default"` or the team's UUID).

Tables with per-team rows: `teams`, `team_settings`, `team_sessions`, `shortlist`.

The `tournaments` table is **shared** — all teams read from the same scraped data. Only shortlist status, notes, and settings are per-team.

The `"default"` team is a synthetic fallback that exists in memory; no row is required in the `teams` table for local development.

## Data pipeline

```
fetch_tournaments()  ←── each collector (ncs / usssa / perfect_game)
        │
        ▼
 upsert_tournaments()  ──► SQLite tournaments table
        │
        │  (async, every 6 h or on demand)
        ▼
 enrich_with_team_lists()  ──► division_teams JSON column
        │
        ▼
 stats.aggregate_team_records()  ──► /api/team-stats
 stats.team_analysis_records()   ──► /api/team-analysis
```

### Collectors

Each collector in `collectors/` implements `fetch_tournaments(radius_miles, target_age, ...)` and returns `list[Tournament]`. The registry (`collectors/registry.py`) maps source names to collector modules.

| Collector | Listing | Team records |
|---|---|---|
| `ncs.py` | HTML scrape of playncs.com event pages | `parse_whos_coming()` per tournament + `ncs_teams.py` scraper |
| `usssa.py` | JSON API (`eventSearchSimpleV11`) | Seeding report API per division |
| `perfect_game.py` | JSON API (`eventGroups`) | HTML team table per division event |

### Team hydration

Records (W-L-T) are fetched in two places:
- **Automatically during refresh** — `enrich_with_team_lists()` called by each collector's `fetch_tournaments()`.
- **On demand** — `POST /api/tournaments/{id}/teams` triggers `services.hydrate_tournament_teams()`, which fills any empty divisions and writes back to `division_teams`.

### Stats engine (`stats.py`)

`aggregate_team_records()` scans all rows in `tournaments` where `division_teams != '{}'`, parses each team's W-L-T record, and aggregates across tournaments per source. The NCS direct-scrape table (`ncs_team_records`) is also merged in so teams appear even before their tournament entries are hydrated.

`team_analysis_records()` does the same but restricts to tournaments on a team's shortlist with status `Interested` or `Registered`.

## Frontend (SPA)

Single-page app served from `static/`. No build step — plain JS loaded as `<script>` tags.

Key files:
- `index.html` — shell, views, templates
- `utils.js` — pure functions (loaded before `app.js`, also used by Vitest unit tests)
- `app.js` — all DOM logic, view routing, fetch calls
- `styles.css` — CSS custom properties themed by `teamThemeTokens()` at runtime

Views: Tournaments · Upcoming · Team Analysis · All Team Stats · Change Log

Theme tokens are generated by `teamThemeTokens()` in `utils.js` from the team's `brand_primary/secondary/accent` settings and applied as CSS variables on `<html>`. Dark/light mode is toggled via a `data-theme` attribute and persisted in `localStorage`.

## Background jobs

When `STAFF_TOOL_ENABLE_HOSTED_JOBS=true` two asyncio tasks start in the lifespan context:

- **`_refresh_loop`** — sleeps for `refresh_cadence_hours` (min 5 min), then calls `services.refresh_sources()` in a thread.
- **`_backup_loop`** — sleeps until UTC midnight, creates a SQLite backup, prunes to 7 most recent.

A threading lock (`_refresh_lock`) prevents overlapping refreshes.

---

## SQLite schema highlights

- `tournaments` — one row per `(source, source_id)`. `division_teams` is a JSON blob written by hydration.
- `shortlist` — `(team_id, tournament_id)` unique; holds status, priority, notes.
- `changes` — immutable log of newly discovered tournaments and field changes.
- `refresh_runs` — start/finish/error log for each source refresh.
- `teams` + `team_settings` — per-team config (age, radius, branding, enabled sources).
- `team_sessions` — bearer token store (hashed); 30-day TTL.
- `ncs_team_records` — records scraped from the NCS Teams listing page, separate from tournament hydration.

Schema migrations use `_ensure_column()` for additive changes — safe for rolling redeploys.

## Security notes

| Concern | Approach |
|---|---|
| Password storage | PBKDF2-SHA256, 100K iterations |
| Session tokens | SHA-256 hashed before storage |
| Cookie flags | `httponly`, `secure` on prod, `samesite=lax` |
| SQL injection | Parameterized queries throughout; dynamic column names are hardcoded |
| XSS | `escapeHtml()` applied before all DOM writes |
| CSRF | `SameSite=Lax` covers browser flows; native API uses bearer, not cookies |
| Error leakage | Exceptions logged server-side; generic messages returned to client |
